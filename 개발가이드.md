# PayFlow 개발 가이드

---

## 1. 프로젝트 개요

PayFlow는 **MSA + 헥사고날 아키텍처 + DDD** 기반의 간편결제 플랫폼이다.

6개의 Bounded Context(Payment, Wallet, Settlement, Member, Merchant, Notification)를 독립 서비스로 분리하고, 도메인 로직을 인프라로부터 격리하여 변경에 강한 구조를 지향한다.

### 핵심 설계 원칙

- **도메인 중심**: 비즈니스 로직은 `domain/` 패키지에 집중. 인프라 의존성 ZERO
- **포트 & 어댑터**: 외부 시스템과의 통신은 Port 인터페이스를 통해 추상화
- **이벤트 기반 통합**: 서비스 간 결합도를 낮추기 위해 Kafka/RabbitMQ 비동기 이벤트 활용
- **장애 격리**: 알림 발송 실패가 결제를 롤백시키지 않음. DLQ로 격리 후 재처리

## 3. 아키텍처 규칙

### 3.1 헥사고날 패키지 구조

각 서비스는 아래 패키지 구조를 따른다.

```
{service}/src/main/java/com/payflow/{service}/
├── domain/                        # 도메인 계층 (핵심)
│   ├── model/                     # Aggregate Root, Entity, Value Object
│   ├── event/                     # Domain Event 정의
│   ├── exception/                 # 도메인 예외
│   └── service/                   # 도메인 서비스 (선택)
│
├── application/                   # 애플리케이션 계층
│   ├── port/
│   │   ├── in/                    # Inbound Port (provided) - UseCase 인터페이스
│   │   └── out/                   # Outbound Port (required) - 인프라 추상화
│   └── service/                   # Application Service (유스케이스 구현)
│
└── adapter/                       # 인프라 계층
    ├── in/
    │   ├── web/                   # REST Controller (Inbound Adapter)
    │   └── messaging/             # Kafka/RabbitMQ Consumer (Inbound Adapter)
    └── out/
        ├── persistence/           # JPA Repository 구현 (Outbound Adapter)
        ├── messaging/             # Kafka/RabbitMQ Producer (Outbound Adapter)
        ├── client/                # 외부 서비스 HTTP Client (Outbound Adapter)
        └── cache/                 # Redis Cache 구현 (Outbound Adapter)
```

### 3.2 Provided / Required 네이밍 규칙

| 구분 | 패키지 | 네이밍 패턴 | 예시 |
|------|--------|-------------|------|
| Inbound Port (Provided) | `application/port/in/` | `{동사}{도메인}UseCase` | `ApprovePaymentUseCase` |
| Outbound Port (Required) | `application/port/out/` | `{동사}{도메인}Port` | `SavePaymentPort`, `LoadMerchantPort` |
| Inbound Adapter | `adapter/in/web/` | `{도메인}Controller` | `PaymentController` |
| Outbound Adapter | `adapter/out/persistence/` | `{도메인}PersistenceAdapter` | `PaymentPersistenceAdapter` |

### 3.3 복잡도별 차등 적용

모든 서비스에 동일한 수준의 헥사고날을 강제하지 않는다.

| 복잡도 | 해당 서비스 | 적용 수준 |
|--------|------------|-----------|
| **높음** | Payment, Settlement | 풀 헥사고날 (Port/Adapter 완전 분리, 도메인 이벤트, Saga) |
| **중간** | Wallet, Merchant | 헥사고날 (Port/Adapter 분리, 도메인 모델) |
| **낮음** | Member, Notification | 경량 헥사고날 (도메인 모델 + 직접 의존 허용) |

## 4. 패키지 규칙

### 4.1 `domain/` -- 외부 의존성 ZERO

```
허용:
  - java.* / javax.*
  - JPA 어노테이션 (@Entity, @Id, @Version 등) -- 유일한 예외
  - 같은 domain/ 내 다른 클래스

금지:
  - Spring 어노테이션 (@Service, @Component, @Autowired 등)
  - application/ 또는 adapter/ 패키지 import
  - 외부 라이브러리 직접 import (Redis, Kafka, HTTP 등)
```

- **Aggregate Root**: 생성자 또는 팩토리 메서드로 생성. 상태 변경은 도메인 메서드를 통해서만
- **Value Object**: 불변(immutable). `equals()`, `hashCode()` 필수 구현
- **Domain Event**: 불변 record 또는 final class. `occurredAt` 필드 필수

### 4.2 `application/` -- Port + Service

```
허용:
  - domain/ 패키지 import
  - port/in/, port/out/ 인터페이스 정의 및 사용
  - Spring @Service, @Transactional

금지:
  - adapter/ 패키지 import (구현체 직접 참조)
  - 인프라 기술 직접 사용 (JPA Repository, RestTemplate 등)
```

- **Application Service**: `@Service` + `@Transactional`. Inbound Port 구현
- **Port Interface**: 인프라 기술에 대한 언급 없이 순수 도메인 용어로 정의

### 4.3 `adapter/` -- 인프라 구현체

```
허용:
  - application/port/ 인터페이스 import 및 구현
  - domain/ 패키지 import
  - Spring, JPA, Kafka, RabbitMQ, Redis 등 인프라 기술 사용

금지:
  - 비즈니스 로직 포함 (로직은 domain/ 또는 application/service/에 위치)
  - 다른 adapter/ 직접 참조 (adapter 간 통신은 Port를 통해)
```


## 7. 커밋 컨벤션

[Conventional Commits](https://www.conventionalcommits.org/) 형식을 따른다.

### 형식

```
<type>(<scope>): <subject>

[body]

[footer]
```

### Type

| Type | 설명 | 예시 |
|------|------|------|
| `feat` | 새로운 기능 | `feat(payment): 부분 취소 기능 추가` |
| `fix` | 버그 수정 | `fix(wallet): 동시 충전 시 잔액 불일치 수정` |
| `docs` | 문서 변경 | `docs: 도메인 모델 문서 업데이트` |
| `test` | 테스트 추가/수정 | `test(settlement): 수수료 계산 단위 테스트 추가` |
| `refactor` | 리팩토링 | `refactor(merchant): CommissionPolicy를 VO로 전환` |
| `chore` | 빌드/설정 변경 | `chore: Gradle 의존성 업데이트` |
| `perf` | 성능 개선 | `perf(settlement): 배치 정산 쿼리 최적화` |

### Scope

서비스명을 scope로 사용한다: `payment`, `wallet`, `settlement`, `member`, `merchant`, `notification`

### 규칙

- subject는 50자 이내, 한국어 또는 영문 (팀 합의에 따라)
- body에 "왜" 변경했는지 설명
- Breaking Change는 footer에 `BREAKING CHANGE:` 표기

## 8. 브랜치 전략

### 브랜치 구조

```
main                              ← 운영 배포 브랜치 (태그로 릴리스)
 └── develop                      ← 개발 통합 브랜치
      ├── feature/payment/partial-cancel     ← 기능 개발
      ├── feature/wallet/charge-limit        ← 기능 개발
      ├── feature/settlement/daily-batch     ← 기능 개발
      └── hotfix/payment/duplicate-fix       ← 긴급 수정
```

### 브랜치 네이밍

```
feature/{service}/{feature-name}    # 기능 개발
hotfix/{service}/{issue-description} # 긴급 수정
release/{version}                    # 릴리스 준비
```

### 워크플로우

1. `develop`에서 `feature/{service}/{feature}` 브랜치 생성
2. 기능 개발 및 테스트
3. PR 생성 → 코드 리뷰 → `develop`에 병합
4. 릴리스 시 `develop` → `release/{version}` → `main` 병합 및 태그

## 9. 테스트 규칙

### 9.1 단위 테스트 (Unit Test)

```
도구: JUnit 5 + Mockito
대상: domain/ 모델, application/ 서비스
Port: Mock 처리 (Mockito @Mock)
실행: ./gradlew test
```

**규칙:**
- 도메인 모델 테스트는 Mock 없이 순수 자바로 작성
- Application Service 테스트는 Outbound Port를 Mock 처리
- 테스트 클래스명: `{클래스명}Test` (예: `PaymentTest`, `ApprovePaymentServiceTest`)
- Given-When-Then 패턴 사용

**예시 구조:**
```java
@ExtendWith(MockitoExtension.class)
class ApprovePaymentServiceTest {

    @Mock
    private SavePaymentPort savePaymentPort;

    @Mock
    private LoadMerchantPort loadMerchantPort;

    @InjectMocks
    private ApprovePaymentService approvePaymentService;

    @Test
    void 결제_승인_성공() {
        // given
        // when
        // then
    }
}
```

### 9.2 통합 테스트 (Integration Test)

```
도구: Spring Boot Test + Testcontainers 2.x
대상: adapter/ 계층 (DB, 메시징, 외부 API)
컨테이너: MySQL, Redis, Kafka, RabbitMQ
실행: ./gradlew integrationTest
```

**규칙:**
- `@Testcontainers` + `@Container`로 인프라 컨테이너 자동 관리
- 테스트 클래스명: `{클래스명}IntegrationTest`
- 테스트 간 데이터 격리 (`@Transactional` 또는 수동 cleanup)

### 9.3 E2E 테스트 (End-to-End Test)

```
도구: REST Assured
대상: API 엔드포인트 전체 흐름
환경: docker compose로 전체 서비스 구동
실행: ./gradlew e2eTest
```

**규칙:**
- 실제 HTTP 요청으로 전체 플로우 검증
- 테스트 클래스명: `{시나리오명}E2ETest`
- 시나리오: 결제 승인 → 정산 생성 → 알림 발송 등 크로스 서비스 플로우

### 9.4 테스트 커버리지 목표

| 계층 | 목표 커버리지 | 필수/권장 |
|------|-------------|-----------|
| `domain/` | 90% 이상 | 필수 |
| `application/service/` | 80% 이상 | 필수 |
| `adapter/` | 70% 이상 | 권장 |

## 10. 코드 리뷰 체크리스트

PR 생성 시 아래 항목을 반드시 확인한다.

### 아키텍처 준수

- [ ] 도메인 로직이 `domain/` 밖으로 유출되지 않았는가?
- [ ] Port 인터페이스가 인프라 기술(JPA, Kafka 등)에 의존하지 않는가?
- [ ] `adapter/`에서 비즈니스 로직을 직접 구현하지 않았는가?
- [ ] `domain/` 패키지에 Spring 어노테이션(`@Service`, `@Component` 등)이 없는가?
- [ ] Aggregate Root를 거치지 않고 내부 Entity를 직접 수정하지 않았는가?

### 도메인 모델

- [ ] Value Object가 불변(immutable)인가?
- [ ] Domain Event에 `occurredAt` 필드가 포함되어 있는가?
- [ ] 상태 전이가 Aggregate Root의 도메인 메서드를 통해 이루어지는가?
- [ ] 잘못된 상태 전이 시 명시적 예외를 던지는가?

### 동시성 / 안정성

- [ ] 지갑 잔액 변경에 낙관적 락(`@Version`)이 적용되어 있는가?
- [ ] 결제 요청에 멱등키(Idempotency Key) 처리가 되어 있는가?
- [ ] 분산락이 필요한 구간에 Redis Lock이 적용되어 있는가?
- [ ] 외부 API 호출에 타임아웃과 서킷브레이커가 설정되어 있는가?

### 이벤트 / 메시징

- [ ] Kafka Producer에 멱등성 설정(`enable.idempotence=true`)이 되어 있는가?
- [ ] RabbitMQ Consumer에 재시도 정책과 DLQ가 구성되어 있는가?
- [ ] 이벤트 발행이 트랜잭션 커밋 이후에 수행되는가? (Transactional Outbox 또는 `@TransactionalEventListener`)

### 테스트

- [ ] 새로운 도메인 로직에 대한 단위 테스트가 추가되었는가?
- [ ] 상태 전이의 정상/예외 케이스가 모두 테스트되었는가?
- [ ] Adapter 변경 시 통합 테스트가 추가/수정되었는가?

### 기타

- [ ] 로그 레벨이 적절한가? (비즈니스 로직 오류: ERROR, 흐름 추적: INFO, 디버그: DEBUG)
- [ ] 민감정보(비밀번호, 계좌번호 등)가 로그에 노출되지 않는가?
- [ ] API 응답에 내부 구현 상세(스택트레이스 등)가 노출되지 않는가?
- [ ] 커밋 메시지가 컨벤션을 따르는가?
