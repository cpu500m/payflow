# PayFlow 도메인 모델

> 핵심 엔티티, Aggregate, 상태 흐름, 도메인 이벤트 정의.
> 비즈니스 용어는 `용어사전.md` 참조.

---

## 1. Bounded Context Map

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          PayFlow Platform                                   │
│                                                                             │
│  ┌──────────────┐    sync HTTP     ┌──────────────┐    sync HTTP           │
│  │              │ ───────────────> │              │ ──────────────>         │
│  │   Payment    │                  │    Wallet    │              ┌────────┐ │
│  │   Context    │ <─ ─ ─ ─ ─ ─ ─  │   Context    │              │Member  │ │
│  │              │   (잔액 응답)     │              │              │Context │ │
│  └──────┬───────┘                  └──────────────┘              └────────┘ │
│         │                                                           ▲       │
│         │ async Kafka                                               │       │
│         │ (PaymentApproved,                                  sync HTTP     │
│         │  PaymentCancelled,                                 (회원 검증)    │
│         │  PaymentFailed)                                           │       │
│         │                                                           │       │
│         ▼                                                           │       │
│  ┌──────────────┐                  ┌──────────────┐         ┌──────┴─────┐ │
│  │              │    sync HTTP     │              │         │            │ │
│  │  Settlement  │ ───────────────> │   Merchant   │         │  Merchant  │ │
│  │   Context    │  (가맹점 조회)    │   Context    │         │  Context   │ │
│  │              │                  │              │         │            │ │
│  └──────────────┘                  └──────────────┘         └────────────┘ │
│         ▲                                                                   │
│         │ async Kafka                                                       │
│         │ (PaymentApproved)                                                │
│         │                                                                   │
│  ┌──────┴───────┐                                                          │
│  │              │  async RabbitMQ                                           │
│  │   Payment    │ ──────────────> ┌──────────────┐                         │
│  │   Context    │                 │ Notification │                         │
│  │              │                 │   Context    │                         │
│  └──────────────┘                 └──────────────┘                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

통신 방식 범례:
  ──────────>  sync HTTP (동기 REST 호출)
  - - - - ->   sync HTTP 응답
  ═══════════> async Kafka (도메인 이벤트)
  ──────────>  async RabbitMQ (알림 메시지)
```

### 통신 요약

| From | To | 방식 | 토픽/큐 | 용도 |
|------|----|------|---------|------|
| Payment | Wallet | sync HTTP | - | 잔액 차감/환불 요청 |
| Payment | Member | sync HTTP | - | 결제 요청 시 회원 유효성 검증 |
| Payment | Merchant | sync HTTP | - | 결제 요청 시 가맹점 유효성 검증 |
| Payment | Settlement | async Kafka | `payment-events` | 결제 승인 이벤트 전파 |
| Payment | Notification | async RabbitMQ | `notification.exchange` | 결제 결과 알림 발송 요청 |
| Settlement | Merchant | sync HTTP | - | 정산 시 가맹점/수수료 정보 조회 |
| Settlement | Notification | async RabbitMQ | `notification.exchange` | 정산 완료 알림 발송 요청 |

## 2. Payment Context

### 2.1 Payment Aggregate

| 필드 | 타입 | 설명 | 제약조건 |
|------|------|------|----------|
| `id` | `PaymentId` (UUID) | 결제 고유 식별자 | PK |
| `orderId` | `String` | 주문 번호 | NOT NULL |
| `memberId` | `MemberId` (UUID) | 결제 회원 ID | NOT NULL, FK |
| `merchantId` | `MerchantId` (UUID) | 결제 가맹점 ID | NOT NULL, FK |
| `amount` | `Money` (VO) | 결제 금액 | NOT NULL, > 0 |
| `method` | `PaymentMethod` | 결제 수단 | NOT NULL |
| `status` | `PaymentStatus` | 결제 상태 | NOT NULL |
| `idempotencyKey` | `String` | 멱등키 | UNIQUE |
| `approvedAt` | `LocalDateTime` | 승인 일시 | NULLABLE |
| `cancelledAt` | `LocalDateTime` | 취소 일시 | NULLABLE |
| `failureReason` | `String` | 실패 사유 | NULLABLE |
| `createdAt` | `LocalDateTime` | 생성 일시 | NOT NULL |
| `updatedAt` | `LocalDateTime` | 수정 일시 | NOT NULL |

### 2.2 PaymentMethod (Enum)

| 코드 | 설명 |
|------|------|
| `CARD` | 신용/체크카드 결제 |
| `WALLET` | PayFlow 지갑 결제 |
| `BANK_TRANSFER` | 계좌이체 결제 |

### 2.3 Money Value Object

| 필드 | 타입 | 설명 |
|------|------|------|
| `amount` | `BigDecimal` | 금액 (소수점 2자리) |
| `currency` | `Currency` | 통화 코드 (기본: KRW) |

- 불변 객체 (Immutable)
- `add()`, `subtract()`, `multiply()`, `isGreaterThan()` 등 연산 메서드 제공
- 음수 금액 생성 시 `InvalidMoneyException` 발생

### 2.4 PaymentStatus 상태 전이

```
                        ┌─────────────┐
                        │  REQUESTED  │
                        └──────┬──────┘
                               │
                 ┌─────────────┼─────────────┐
                 │ approve()   │             │ fail()
                 ▼             │              ▼
          ┌──────────┐        │       ┌──────────┐
          │ APPROVED │        │       │  FAILED  │
          └─────┬────┘        │       └──────────┘
                │             │
                │ cancel()    │
                ▼             │
          ┌──────────┐        │
          │CANCELLED │        │
          └──────────┘        │
                              │
  허용되는 전이:
  REQUESTED → APPROVED  : 결제 승인 성공
  REQUESTED → FAILED    : 결제 승인 실패 (잔액 부족, PG 오류 등)
  APPROVED  → CANCELLED : 승인 후 결제 취소

  금지되는 전이:
  FAILED    → (any)     : 실패는 최종 상태
  CANCELLED → (any)     : 취소는 최종 상태
  APPROVED  → FAILED    : 승인 후 실패 불가 (취소만 가능)
```

| 필드 | 타입 | 설명 | 제약조건 |

| 필드 | 타입 | 설명 | 제약조건 |
|------|------|------|----------|
| `id` | `TransactionId` (UUID) | 거래 고유 식별자 | PK |
| `walletId` | `WalletId` (UUID) | 소속 지갑 ID | FK |
| `type` | `TransactionType` | 거래 유형 | NOT NULL |
| `amount` | `Money` (VO) | 거래 금액 | NOT NULL, > 0 |

### 3.3 TransactionType (Enum)

| 코드 | 설명 | 잔액 영향 |
|------|------|-----------|
| `CHARGE` | 충전 | + (증가) |
| `DEDUCT` | 차감 | - (감소) |
| `REFUND` | 환불 | + (증가) |

### 3.4 잔액 흐름 다이어그램

```
  [외부 결제 수단]           [Wallet]              [Payment]
        │                      │                      │
        │   충전 요청           │                      │
        │ ──────────────────> │                      │
        │                     │ balance += amount     │
        │                     │ Transaction(CHARGE)   │
        │                     │                      │
        │                     │    차감 요청           │
        │                     │ <──────────────────── │
        │                     │ balance -= amount     │
        │                     │ Transaction(DEDUCT)   │
        │                     │                      │
        │                     │    환불 요청           │
        │                     │ <──────────────────── │
        │                     │ balance += amount     │
        │                     │ Transaction(REFUND)   │
        │                     │                      │

  잔액 불변식: balance >= 0 (차감 시 잔액 부족이면 InsufficientBalanceException)
  동시성 제어: @Version 낙관적 락 → OptimisticLockException 발생 시 재시도
```

| `status` | `SettlementStatus` | 정산 상태 | NOT NULL |

### 4.2 SettlementTarget Entity

| 필드 | 타입 | 설명 | 제약조건 |
|------|------|------|----------|
| `id` | `UUID` | 정산 대상 고유 ID | PK |
| `settlementId` | `SettlementId` (UUID) | 소속 정산 ID | FK |
| `paymentId` | `PaymentId` (UUID) | 원본 결제 ID | NOT NULL |
| `paymentAmount` | `Money` (VO) | 결제 금액 | NOT NULL |
| `commissionAmount` | `Money` (VO) | 해당 건 수수료 | NOT NULL |
| `netAmount` | `Money` (VO) | 해당 건 순정산금액 | NOT NULL |
| `createdAt` | `LocalDateTime` | 생성 일시 | NOT NULL |

### 4.3 CommissionPolicy Value Object

| 필드 | 타입 | 설명 |
|------|------|------|
| `rate` | `BigDecimal` | 수수료율 (예: 0.035 = 3.5%) |
| `baseFee` | `Money` | 건당 기본 수수료 |

- 수수료 계산: `commission = paymentAmount * rate + baseFee`
- 불변 객체

### 4.4 SettlementStatus 상태 전이

```
          ┌──────────┐
          │ PENDING  │  ← 결제 승인 이벤트 수신 시 자동 생성
          └─────┬────┘
                │ calculate()
                ▼
        ┌────────────┐
        │ CALCULATED │  ← 배치에서 수수료 산출 완료
        └──────┬─────┘
               │ confirm()
               ▼
        ┌────────────┐
        │ CONFIRMED  │  ← 관리자 확정 또는 자동 확정
        └──────┬─────┘
               │ pay()
               ▼
        ┌────────────┐
        │    PAID    │  ← 가맹점 계좌로 실제 이체 완료
        └────────────┘

  허용되는 전이:
  PENDING    → CALCULATED : 배치 정산 산출
  CALCULATED → CONFIRMED  : 정산 확정
  CONFIRMED  → PAID       : 지급 완료

  역방향 전이 불가 (단, CALCULATED → PENDING 은 재산출 시 허용)
```

### 4.5 정산 배치 흐름

```
  ┌───────────────────────────────────────────────────────────────┐
  │ Settlement Batch (매일 02:00 KST 실행)                         │
  │                                                               │
  │  1. Kafka에서 수신한 PaymentApproved 이벤트 기반으로             │
  │     PENDING 상태의 SettlementTarget 목록 조회                   │
  │                                                               │
  │  2. 가맹점별로 그룹핑                                           │
  │                                                               │
  │  3. Merchant Context에서 CommissionPolicy 조회 (sync HTTP)     │
  │                                                               │
  │  4. 수수료 계산: commission = amount * rate + baseFee           │
  │                                                               │
  │  5. Settlement 생성 (status = CALCULATED)                      │
  │     - totalAmount = SUM(paymentAmounts)                        │
  │     - commissionAmount = SUM(commissions)                      │
  │     - netAmount = totalAmount - commissionAmount               │
  │                                                               │
  │  6. 자동 확정 조건 충족 시 → CONFIRMED                          │
  │                                                               │
  │  7. 지급 처리 → PAID, RabbitMQ로 알림 발송                      │
  └───────────────────────────────────────────────────────────────┘
```

### 5.2 MerchantStatus 상태 전이

```
          ┌──────────┐
          │ PENDING  │  ← 가맹점 등록 신청
          └─────┬────┘
                │ approve()
                ▼
          ┌──────────┐
          │  ACTIVE  │ ←─────────────┐
          └─────┬────┘               │
                │ suspend()    reactivate()
                ▼                    │
          ┌──────────┐               │
          │SUSPENDED │ ──────────────┘
          └─────┬────┘
                │ terminate()
                ▼
        ┌────────────┐
        │ TERMINATED │  ← 최종 상태 (복구 불가)
        └────────────┘

  허용되는 전이:
  PENDING   → ACTIVE     : 심사 승인
  ACTIVE    → SUSPENDED  : 서비스 정지
  SUSPENDED → ACTIVE     : 정지 해제 (재활성화)
  SUSPENDED → TERMINATED : 최종 해지
  ACTIVE    → TERMINATED : 즉시 해지

  PENDING   → TERMINATED : 심사 거절에 의한 해지 (reject)
  TERMINATED → (any)     : 해지는 최종 상태, 복구 불가
```

### 6.3 MemberStatus 상태 전이

```
          ┌──────────┐
          │  ACTIVE  │ ←─────────────┐
          └─────┬────┘               │
           │         │         reactivate()
    withdraw()  toDormant()          │
           │         │               │
           │         ▼               │
           │   ┌──────────┐          │
           │   │ DORMANT  │ ─────────┘
           │   └─────┬────┘
           │         │ withdraw()
           ▼         ▼
        ┌────────────┐
        │ WITHDRAWN  │  ← 최종 상태
        └────────────┘

  허용되는 전이:
  ACTIVE  → DORMANT   : 1년 미로그인 시 배치 전환
  ACTIVE  → WITHDRAWN : 회원 직접 탈퇴
  DORMANT → ACTIVE    : 재인증 후 복귀
  DORMANT → WITHDRAWN : 휴면 상태에서 탈퇴

  WITHDRAWN → (any)   : 탈퇴는 최종 상태, 복구 불가
  탈퇴 시: 개인정보 마스킹 처리 (이름, 이메일, 연락처)
```

## 7. Notification Context

### 7.1 Notification Model

| 필드 | 타입 | 설명 | 제약조건 |
|------|------|------|----------|
| `id` | `UUID` | 알림 고유 식별자 | PK |
| `recipientId` | `UUID` | 수신자 ID (회원 또는 가맹점) | NOT NULL |
| `recipientType` | `RecipientType` | 수신자 유형 (`MEMBER`, `MERCHANT`) | NOT NULL |
| `type` | `NotificationType` | 알림 유형 | NOT NULL |
| `channel` | `NotificationChannel` | 발송 채널 (`EMAIL`, `SMS`, `PUSH`) | NOT NULL |
| `title` | `String` | 알림 제목 | NOT NULL |
| `content` | `String` | 알림 본문 | NOT NULL |
| `status` | `NotificationStatus` | 발송 상태 (`PENDING`, `SENT`, `FAILED`) | NOT NULL |
| `retryCount` | `Integer` | 재시도 횟수 | DEFAULT 0 |
| `sentAt` | `LocalDateTime` | 발송 일시 | NULLABLE |
| `failedReason` | `String` | 실패 사유 | NULLABLE |
| `createdAt` | `LocalDateTime` | 생성 일시 | NOT NULL |

### 7.2 NotificationType (Enum)

| 코드 | 설명 | 수신자 |
|------|------|--------|
| `PAYMENT_APPROVED` | 결제 승인 알림 | 회원 |
| `PAYMENT_CANCELLED` | 결제 취소 알림 | 회원 |
| `PAYMENT_FAILED` | 결제 실패 알림 | 회원 |
| `WALLET_CHARGED` | 지갑 충전 완료 알림 | 회원 |

### 7.3 RabbitMQ 토폴로지

```
  Producer (Payment, Settlement, Wallet)
       │
       ▼
  ┌─────────────────────────────────────────┐
  │  Exchange: notification.exchange         │
  │  Type: topic                            │
  │  Durable: true                          │
  └────────┬──────────────┬─────────────────┘
           │              │
  routing: │              │ routing:
  payment.*│              │ settlement.*
           │              │
           ▼              ▼
  ┌────────────────┐  ┌─────────────────────┐
  │ Queue:         │  │ Queue:              │
  │ payment        │  │ settlement          │
  │ .notification  │  │ .notification       │
  │ .queue         │  │ .queue              │
  └───────┬────────┘  └──────────┬──────────┘
          │                      │
          │  (3회 재시도 실패)     │  (3회 재시도 실패)
          ▼                      ▼
  ┌────────────────┐  ┌─────────────────────┐
  │ DLQ:           │  │ DLQ:                │
  │ payment        │  │ settlement          │
  │ .notification  │  │ .notification       │
  │ .dlq           │  │ .dlq               │
  └────────────────┘  └─────────────────────┘

  재시도 정책:
  - 최대 3회 재시도
  - 지수 백오프: 1초 → 2초 → 4초
  - 3회 실패 시 DLQ로 이동
  - DLQ 메시지는 관리자 수동 재처리 또는 모니터링 대시보드에서 확인
```

## 8. 서비스 간 통신 정리

| From | To | 방식 | 토픽/큐/엔드포인트 | 이유 |
|------|----|------|-------------------|------|
| Payment | Wallet | sync HTTP | `POST /api/v1/wallets/deduct` | 결제 시 즉시 잔액 차감이 필요하므로 동기 호출. 실패 시 결제도 실패 처리 |
| Payment | Wallet | sync HTTP | `POST /api/v1/wallets/refund` | 취소 시 즉시 잔액 복원이 필요하므로 동기 호출 |
| Payment | Member | sync HTTP | `GET /api/v1/members/{id}` | 결제 요청 시 회원 존재 및 상태 검증. 즉시 응답 필요 |
| Payment | Merchant | sync HTTP | `GET /api/v1/merchants/{id}` | 결제 요청 시 가맹점 유효성 및 활성 상태 검증 |
| Payment | Settlement | async Kafka | `payment-events` (PaymentApproved) | 정산은 결제 완료 후 비동기로 처리해도 되며, 느슨한 결합 유지 |
| Payment | Notification | async RabbitMQ | `notification.exchange` | 알림 발송 실패가 결제를 롤백시키면 안 되므로 비동기. 재시도 + DLQ 보장 |
| Settlement | Merchant | sync HTTP | `GET /api/v1/merchants/{id}/commission` | 정산 산출 시 수수료 정책을 즉시 조회해야 하므로 동기 호출 |
| Settlement | Notification | async RabbitMQ | `notification.exchange` | 정산 완료 알림은 비동기 발송으로 충분 |
| Wallet | Notification | async RabbitMQ | `notification.exchange` | 충전 완료 알림은 비동기 발송으로 충분 |
