# PayFlow 용어사전

> PayFlow 간편결제 플랫폼에서 사용되는 도메인 용어 정의.
> 엔티티 상세는 `도메인모델.md` 참조.

---

## 1. 결제 (Payment)

| 한국어 | 영문 (코드) | 설명 |
|--------|-------------|------|
| 결제 | Payment | 회원이 가맹점에 대금을 지불하는 행위. PayFlow의 핵심 Aggregate Root |
| 결제 승인 | Payment Approval | PG사 또는 지갑을 통해 결제 금액이 확정되어 승인된 상태. `PaymentApproved` 이벤트 발행 |
| 결제 취소 | Payment Cancellation | 승인된 결제를 전액 또는 부분 취소하는 행위. `PaymentCancelled` 이벤트 발행 |
| 결제 수단 | PaymentMethod | 결제에 사용되는 수단. `CARD`(신용/체크카드), `WALLET`(PayFlow 지갑), `BANK_TRANSFER`(계좌이체) |
| 결제 상태 | PaymentStatus | 결제의 생명주기 상태 코드 |
| - 요청됨 | `REQUESTED` | 결제가 생성되어 승인 대기 중인 초기 상태 |
| - 승인됨 | `APPROVED` | PG사 또는 지갑에서 결제가 정상 승인된 상태 |
| - 취소됨 | `CANCELLED` | 승인 후 전액 취소가 완료된 상태 |
| - 실패 | `FAILED` | PG 통신 오류, 잔액 부족 등으로 결제가 실패한 최종 상태 |
| 중복 결제 | Duplicate Payment | 동일 주문에 대해 결제가 2회 이상 승인되는 장애 상황. 멱등키(Idempotency Key)로 방지 |
| 분산락 | Distributed Lock | Redis 기반으로 동시 결제 요청의 동시성을 제어하는 메커니즘. Redisson `RLock` 사용 |
| 멱등키 | Idempotency Key | 클라이언트가 생성하는 고유 키. 동일 키의 재요청 시 기존 결과를 반환하여 중복 처리 방지 |
| 결제 금액 | Money (VO) | 금액(`amount`)과 통화(`currency`)를 캡슐화한 Value Object. 불변 객체 |

## 2. 지갑 (Wallet)

| 한국어 | 영문 (코드) | 설명 |
|--------|-------------|------|
| 지갑 | Wallet | 회원별로 1:1 매핑되는 선불 충전식 전자지갑. Aggregate Root |
| 잔액 | Balance | 지갑에 보유 중인 현재 금액. `Money` VO로 표현 |
| 충전 | Charge | 외부 결제 수단(카드, 계좌이체)으로 지갑에 금액을 추가하는 행위 |
| 차감 | Deduct | 결제 시 지갑 잔액에서 결제 금액을 빼는 행위 |
| 환불 | Refund | 결제 취소 시 차감된 금액을 지갑으로 되돌리는 행위 |
| 거래 원장 | Transaction Ledger | 지갑의 모든 입출금 내역을 시간순으로 기록한 불변 로그 |
| 거래 유형 | TransactionType | 거래 원장에 기록되는 거래의 종류 |
| - 충전 | `CHARGE` | 외부에서 지갑으로 금액이 유입된 거래 |
| - 차감 | `DEDUCT` | 결제로 인해 지갑에서 금액이 유출된 거래 |
| - 환불 | `REFUND` | 결제 취소로 인해 지갑으로 금액이 복원된 거래 |
| 낙관적 락 | Optimistic Lock | 지갑 잔액 업데이트 시 `version` 필드를 이용한 동시성 제어. JPA `@Version` 활용 |
| 거래 참조 ID | Reference ID | 거래를 발생시킨 원인(결제 ID 등)을 추적하기 위한 외부 참조 식별자 |

## 3. 정산 (Settlement)

| 한국어 | 영문 (코드) | 설명 |
|--------|-------------|------|
| 정산 | Settlement | 가맹점에 지급할 금액을 산출하고 지급하는 프로세스. Aggregate Root |
| 수수료 | Commission | 결제 금액에서 PayFlow가 차감하는 서비스 이용 대가 |
| 수수료 정책 | CommissionPolicy (VO) | 가맹점별로 적용되는 수수료 계산 규칙. 수수료율(`rate`)과 기본 수수료(`baseFee`)로 구성 |
| 수수료율 | Commission Rate | 결제 금액 대비 수수료 비율. 가맹점 계약에 따라 차등 적용 (예: 3.5%) |
| 정산 주기 | Settlement Cycle | 정산 금액을 가맹점에 지급하는 반복 주기. `DAILY`, `WEEKLY`, `MONTHLY` |
| 정산 상태 | SettlementStatus | 정산의 처리 단계 코드 |
| - 대기 | `PENDING` | 정산 대상으로 등록되었으나 아직 계산되지 않은 상태 |
| - 산출 완료 | `CALCULATED` | 수수료 차감 후 순정산금액이 산출된 상태 |
| - 확정 | `CONFIRMED` | 관리자 또는 시스템에 의해 정산이 최종 확정된 상태 |
| - 지급 완료 | `PAID` | 가맹점 계좌로 정산 금액이 실제 이체된 상태 |
| 정산 대상 | SettlementTarget | 정산에 포함되는 개별 결제 건의 상세 내역 엔티티 |
| 순정산금액 | Net Settlement Amount | 총 결제 금액에서 수수료를 차감한 가맹점 실수령 금액 |
| 정산 배치 | Settlement Batch | 정산 주기에 따라 대량의 정산 건을 일괄 처리하는 스케줄링 작업 |

## 4. 가맹점 (Merchant)

| 한국어 | 영문 (코드) | 설명 |
|--------|-------------|------|
| 가맹점 | Merchant | PayFlow와 계약을 맺고 결제 서비스를 이용하는 사업자. Aggregate Root |
| 가맹점 상태 | MerchantStatus | 가맹점의 계약 및 서비스 이용 상태 |
| - 심사 중 | `PENDING` | 가맹점 등록 신청 후 심사 대기 중인 상태 |
| - 활성 | `ACTIVE` | 심사 승인 후 정상 서비스 이용 가능 상태 |
| - 정지 | `SUSPENDED` | 약관 위반 등으로 일시적으로 서비스 이용이 중단된 상태 |
| - 해지 | `TERMINATED` | 계약이 최종 해지되어 복구 불가능한 상태 |
| 사업자번호 | Business Registration Number | 가맹점 사업자의 고유 등록번호 (10자리). 중복 등록 불가 |
| 수수료율 | Commission Rate | 해당 가맹점에 적용되는 결제 수수료 비율 |
| 정산 계좌 | Settlement Account | 정산 금액이 입금되는 가맹점의 은행 계좌 정보 |
| 가맹점 ID | Merchant ID | 시스템 내부에서 가맹점을 식별하는 UUID |

## 5. 회원 (Member)

| 한국어 | 영문 (코드) | 설명 |
|--------|-------------|------|
| 회원 | Member | PayFlow 서비스에 가입한 개인 사용자. Aggregate Root |
| 회원 상태 | MemberStatus | 회원의 서비스 이용 상태 |
| - 활성 | `ACTIVE` | 정상 서비스 이용 가능 상태 |
| - 휴면 | `DORMANT` | 장기간(1년) 미로그인으로 전환된 상태. 재인증 시 `ACTIVE` 복귀 |
| - 탈퇴 | `WITHDRAWN` | 회원 탈퇴 처리된 최종 상태. 개인정보 삭제/마스킹 수행 |
| 이메일 | Email (VO) | 회원 로그인 ID이자 알림 수신 주소. Value Object로 형식 검증 내장 |
| 비밀번호 | Password | BCrypt로 해싱되어 저장되는 회원 인증 수단 |
| 회원 ID | Member ID | 시스템 내부에서 회원을 식별하는 UUID |

## 6. 알림 (Notification)

| 한국어 | 영문 (코드) | 설명 |
|--------|-------------|------|
| 알림 | Notification | 회원 또는 가맹점에게 발송되는 시스템 메시지 |
| 알림 유형 | NotificationType | 알림 발송 트리거가 되는 도메인 이벤트 유형 |
| - 결제 승인 | `PAYMENT_APPROVED` | 결제가 정상 승인되었을 때 회원에게 발송 |
| - 결제 취소 | `PAYMENT_CANCELLED` | 결제가 취소되었을 때 회원에게 발송 |
| - 결제 실패 | `PAYMENT_FAILED` | 결제가 실패했을 때 회원에게 발송 |
| - 충전 완료 | `WALLET_CHARGED` | 지갑 충전이 완료되었을 때 회원에게 발송 |
| - 정산 완료 | `SETTLEMENT_PAID` | 정산 금액이 지급 완료되었을 때 가맹점에게 발송 |
| DLQ | Dead Letter Queue | 알림 발송 실패 메시지가 격리되는 RabbitMQ 큐. 수동 재처리 또는 모니터링 대상 |
| 재시도 | Retry | 알림 발송 실패 시 지수 백오프(exponential backoff)로 최대 3회 재시도하는 정책 |
| 알림 채널 | Notification Channel | 알림이 전달되는 매체. `EMAIL`, `SMS`, `PUSH` |

## 7. 공통 인프라

| 한국어 | 영문 (코드) | 설명 |
|--------|-------------|------|
| MSA | Microservices Architecture | 서비스를 독립적으로 배포 가능한 작은 단위로 분리하는 아키텍처 패턴. PayFlow는 6개 서비스로 구성 |
| 헥사고날 아키텍처 | Hexagonal Architecture | 도메인 로직을 중심에 놓고, 외부 인프라(DB, 메시징 등)를 어댑터로 분리하는 아키텍처. Ports & Adapters 패턴 |
| Bounded Context | Bounded Context | DDD에서 도메인 모델이 일관된 의미를 갖는 경계. PayFlow는 Payment, Wallet, Settlement, Member, Merchant, Notification 6개 BC로 구성 |
| Aggregate Root | Aggregate Root | 트랜잭션 일관성을 보장하는 엔티티 클러스터의 진입점. 외부에서는 반드시 Aggregate Root를 통해서만 내부 엔티티에 접근 |
| Domain Event | Domain Event | 도메인에서 발생한 의미 있는 사건을 표현하는 불변 객체. Kafka를 통해 BC 간 비동기 전파 |
| Inbound Port | Inbound Port (Provided) | 외부에서 도메인을 호출하기 위해 도메인이 제공하는 인터페이스. `application/port/in/` 패키지에 위치 |
| Outbound Port | Outbound Port (Required) | 도메인이 외부 인프라를 사용하기 위해 요구하는 인터페이스. `application/port/out/` 패키지에 위치 |
| KRaft | KRaft (Kafka Raft) | ZooKeeper 없이 Kafka 자체 합의 프로토콜로 메타데이터를 관리하는 모드. PayFlow는 Kafka 4.x KRaft 사용 |
| 이벤트 소싱 | Event Sourcing | 상태 변경을 이벤트의 시퀀스로 저장하여, 이벤트를 재생(replay)해 현재 상태를 복원하는 패턴 |
| 사가 패턴 | Saga Pattern | 분산 트랜잭션을 로컬 트랜잭션의 연쇄로 처리하고, 실패 시 보상 트랜잭션으로 롤백하는 패턴 |
| CQRS | Command Query Responsibility Segregation | 명령(쓰기)과 조회(읽기) 모델을 분리하는 패턴. 정산 조회 등 읽기 성능 최적화에 활용 |
| 멱등성 | Idempotency | 동일 요청을 여러 번 수행해도 결과가 동일한 성질. 결제 API의 핵심 요구사항 |
| 보상 트랜잭션 | Compensating Transaction | 사가 패턴에서 이전 단계의 커밋을 논리적으로 되돌리기 위해 실행하는 역방향 트랜잭션 |
